Cortex-A7 Startup Example Code for Arm Compiler for Embedded 6 - Arm®Development Studio

This example shows the bare-metal startup code for the Cortex-A7 processor, including vector table, exception handlers, MMU, cache, FPU/NEON initialization, and is illustrated by a simple semihosted "sorts" example application.

Purpose and scope

This example shows the bare-metal startup code for the Cortex-A7 processor, including vector table, exception handlers, MMU, cache, FPU/NEON initialization. A semihosted "sorts" example is provided in sorts.c for demonstration purposes, that can be replaced by your own application code. This example can be used as a framework on which to build your own C/C++ applications.
This example is intended to be built with Arm Compiler for Embedded 6. If you wish to modify and rebuild the example, you must use Arm Compiler for Embedded 6 to rebuild it.
This example is intended to run on the ST STM32MP15x Evaluation board (available from ST Microelectronics).
A ready-made launch configuration startup_Cortex-A7_STM32MP15x.launch is provided.
To run on the ST STM32MP15x you will need:
  • An ST STM32MP15x Evaluation board ('the target') and a suitable power supply for it.
  • DSTREAM or ULINK debug hardware and a suitable power supply for it, and a JTAG cable to connect it to the target.
  • A serial terminal emulator such as the Terminal view in Development Studio, PuTTY, TeraTerm, or Minicom for Linux, connected via a serial cable to your platform. This may be needed when running the example on real target hardware, to monitor the Boot Loader and provide a serial terminal interface. This is not needed if using a software model as that provides its own terminal. To open the Terminal view in Development Studio, select Window → Show View → Other... to open the Show View dialog box, then expand the Terminal group and select Terminal, and click OK. To configure the terminal settings, click on Open a Terminal in the Terminal view, then select the required connection type (for example, Serial Terminal), then enter the required settings (for example, 38400 baud, 8 bits, no parity, 1 stop bit) and click OK.

Building the example

This example can be built with Arm Compiler for Embedded 6 using the supplied Eclipse project, or directly on the command-line with the supplied Makefile.
The Eclipse project is a managed builder project, that creates a makefile in the /Debug directory.
This executable has an application base address 0x10000000.
This example depends on semihosting support being provided by the debug system. The Debugger enables semihosting automatically if either symbols __auto_semihosting or __semihosting_library_function are present in an image. Arm Compiler for Embedded 6 and Arm Compiler 5 both add __semihosting_library_function automatically to an image at link time if that image uses any semihosting-using functions. If compiling with GCC, use an alias symbol definition such as void __auto_semihosting(void) __attribute__ ((alias("main"))); or turn on semihosting support in the Debugger explicitly with set semihosting enabled on.

Building on the command-line

To build the example on the command-line with the supplied make utility:
  • On Windows, open an Arm DS Command Prompt from the Start menu, run the select_toolchain utility, and select Arm Compiler for Embedded 6 from the list
  • On Linux, run the suite_exec utility with the --toolchain option to select the compiler and start a shell configured for the Development Studio environment, for example: ~/developmentstudio/bin/suite_exec --toolchain "Arm Compiler for Embedded 6" bash
Then navigate to the ...\startup_Cortex-A7_STM32MP15x directory, and type:
make
The usual make rules: clean, all and rebuild are provided in the makefile.

Building within the IDE

To build the supplied projects within the IDE:
  1. In the Project Explorer view, select the project you want to build.
  2. Select Project → Build Project.

Running the example on the STM32MP15x

  1. NOTE: on this target, the Cortex-A7 will enter 'Normal' mode under normal SDCard boot circumstances. Therefore it is recommended that the board is booted into 'Engineering' mode so that the Cortex-A7's are powered-up and remain in the Secure state. 'Engineering' boot mode can be selected by setting SW1[1..3] to OFF OFF ON
  2. Power-up the STM32MP15x
  3. Select Run → Debug Configurations....
  4. Select startup_Cortex-A7_STM32MP15x from the list of Generic Arm C/C++ Application configurations.
  5. In the Connections panel, enter the USB: or TCP: IP address or name of your DSTREAM or ULINK unit in the Debug Hardware Address field, or click on Browse to select one from a list, otherwise an error will be reported: Launch configuration has errors: Configuration for connection type 'Bare Metal Debug' is not valid - Connection cannot be empty .
  6. Click on Debug to start debugging. The example executable will be downloaded to the target, and the program counter PC will be set to the entry point of the image.
  7. Run the executable (press F8). Text output appears in the App Console view.

Debugging the reset handler

To debug the reset handler, first disconnect any existing debug session, then load the executable (startup_Cortex-A7_STM32MP15x.axf) in the same way as before, selecting "Debug from entry point" in the Debugger tab.
The Debugger will download the program's code and data sections to the target, and set the program counter PC to the entry point of the image, at the reset entry in the vector table, on the LDR PC, Reset_Addr instruction in startup.S at address 0x10000000.
  1. In the Registers view, expand Core to see the core registers.
  2. Execute the LDR PC, Reset_Addr instruction by single-stepping (press F5) into Reset_Handler. Notice the Program Counter (PC) change in the Core registers view.
  3. The first instructions inside Reset_Handler (MRC, BIC, MCR) perform a read/modify/write of CP15 Control Register to disable the caches and MMU. In the Registers view, expand CP15, then expand System, then expand SCTLR. Single-step (press F5) through these instructions, and see the I, C and M bits being cleared (if they were previously set).
  4. The next instructions (MRC, ORR, MCR) set the SMP bit to enable coherent requests to the processor.
  5. The next instruction (LDR) initializes the Supervisor mode stack. In the Registers view, expand Core and see SP being initialized. Single-step (press F5) through this instruction.
  6. The next instructions (MOV, MCR) invalidate the Data and Instruction TLBs. Single-step (press F5) through these instructions.
  7. The next instructions (MOV, MCR) invalidate the Instruction cache and flushes the BTAC. Single-step (press F5) through these instructions.
  8. The next instructions (LDR, MCR) set the Vector Base Address Register (VBAR) to point to this application's vector table. In the Registers view, inside the expanded CP15, expand SecurityExtensions. Single-step (press F5) through these instructions, and see the VBAR being set to 0x80000000.
  9. The next blocks of assembler instructions perform cache invalidation, MMU configuration, page-table generation (the page-table configuration will be studied in the next section), and if enabled, switch on the VFP and NEON hardware. Single-step through the first few of these and watch their effects on the CP15 registers.
  10. Set a breakpoint on LDR r12,=__main and run to it (press F8). This prepares for a (possibly long) branch to the C library initialization code at __main(). Single-step and observe r12 being loaded. The next instructions enable the MMU. Single-step through these instructions and see the CP15 System Control Register being updated. The MMU is now enabled and virtual to physical address translations will start occuring. Finally, the code performs a Branch Exchange (BX) to __main().
  11. Set a breakpoint on the first instruction in the enable_caches() function, and run to it (press F8). This function called from main() enables the caches. Single-step through these instructions and see the CP15 System Control Register being updated, until BX lr is reached. Single-step again to return back to the function call in main() in sorts.c.
  12. Delete all breakpoints, and continue running (press F8) until the application terminates normally.

Viewing the MMU and page-table configuration

Load the executable in the same way as before, but this time select "Debug from symbol" main in the Debugger tab.
The Debugger will download the program's code and data sections to the target, and run to main() inside main.c. Startup code executed earlier by CPU0 will have already configured the page-tables and enabled the MMU. To view the MMU and page-table configuration:
  1. Open the MMU/MPU view with Window → Show View → MMU/MPU.
  2. In the MMU/MPU view, select the Memory Map tab. This gives a top-level view of the virtual memory layout, by combining translation table entries that map to contiguous regions of physical memory with common memory type, cacheability, shareability and access attributes.
  3. In the MMU/MPU view, select the Tables tab, then expand TTBR0. The lower pane shows the TTBR0 page-tables located at SP:0x10020000. Notice that for this example, each entry is "flat-mapped" - an input (virtual) address 0x10000000 maps to the same output (physical) address 0x10000000. Also in this example, the entire memory map is configured as 'strongly-ordered' memory, except for the 1MB section based at 0x10000000 that is setup specially as cacheable 'normal' memory for executable application code.
  4. In the Commands view, enter mmu print. This gives a similar output to the above:

  5. In the MMU/MPU view, select the Translation tab. This allows you to see which physical address is mapped to a particular virtual address, and vice-versa. For example, enter 0x10000000, select Virtual to Physical then press Translate. The lower pane shows the translated address. For this example, the translated address is 0x10000000 because this example uses "flat-mapping". Now select Physical to Virtual then press Translate. Again the result is 0x10000000.
  6. In the Commands view, enter mmu translate 0x10000000 then press Submit. This gives a similar output to the Virtual to Physical address translation above. To translate Physical to Virtual addresses, enter, for example, mmu translate SP:0x10000000.
  7. Disconnect.
